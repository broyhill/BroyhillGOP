# What does BROYHILLGOP BRAIN diagram represent

The “BROYHILLGOP BRAIN” diagram is describing a conceptual **architecture** for an AI‑driven campaign decision system, not a biological brain. It represents how donor and campaign data flow through several layers—perception, reasoning, decision, execution, and learning—to automatically plan, optimize, and improve political fundraising and outreach.[^2_1]

### What each layer means

- Perception Layer: Ingests inputs like donor behavior (opens, clicks, donations), candidate actions (new content, events), external triggers (news, polls), and performance metrics (ROI, response rates, costs).[^2_1]
- Reasoning Layer: Uses models and rules (matcher, predictor, optimizer plus a “personality engine”) to score audiences, forecast outcomes, choose channels, and keep messages on‑brand and within defined tone/ethics constraints.[^2_1]
- Decision Layer: Turns that reasoning into concrete choices: who to contact, when, with what message, through which channel, and at what spend level.[^2_1]
- Execution Layer: Pushes those decisions into operational systems like email, SMS, phone, mail, social, events, voice, and video tools to actually run the outreach.[^2_1]
- Learning Layer: Collects results (opens, clicks, donations, unsubscribes, ROI), updates ML models, refines decision rules, and shares learnings across candidates, closing the feedback loop.[^2_1]


### Overall meaning

Together, the diagram is a blueprint for a closed‑loop, data‑driven campaign “brain” that continuously senses what’s happening, reasons about the best actions, executes them across channels, and learns from the outcomes to get smarter over time.[^2_1]

| Factor | Max Points | How Calculated |
| :-- | :-- | :-- |

# BROYHILLGOP BRAIN: Comprehensive Gap Analysis \& Definitions Required for Production

Based on a systematic audit of all 76 ecosystem Python files, 60+ database schema SQL files, and supporting documentation in the repository, here are the in-depth definitions needed to make this platform come alive, organized by critical system layer.

***

## 1. INTER-ECOSYSTEM EVENT BUS (Missing Entirely)

The single largest gap across the entire codebase. Every ecosystem operates as a standalone Python class with its own database connection, but **no message broker or event bus exists to connect them**.[^7_1]

**Definitions needed:**

- **Event Schema Contract**: A formal schema (protobuf, JSON Schema, or Avro) defining every event type. E.g., `donation.completed` from E02 must carry `{donor_id, candidate_id, amount, election_type}` so E01 (Donor Intelligence) can trigger grade recalculation, E20 (Intelligence Brain) can update models, and E40 (Automation) can fire workflows.
- **Pub/Sub Topology Map**: Which ecosystem publishes which events, and which subscribes. Currently E02 has `_trigger_grade_update()` that just logs a message. E51 Nexus references E20 for GO/NO-GO decisions but calls them directly via function, not events. Each of these cross-ecosystem calls needs to be defined as either synchronous RPC or async event.[^7_2][^7_1]
- **Message Broker Selection**: Redis Streams, RabbitMQ, or PostgreSQL LISTEN/NOTIFY. The codebase uses PostgreSQL everywhere, so pg_notify is the lowest-friction choice, but needs formal channel definitions.
- **Dead Letter Queue Policy**: What happens when E30 Email publishes a `campaign.sent` event but E06 Analytics is down? Retry count, backoff intervals, and alerting thresholds.

***

## 2. AUTHENTICATION \& AUTHORIZATION LAYER (Missing Entirely)

No ecosystem defines user roles, permissions, or authentication.[^7_3]

**Definitions needed:**

- **Role Taxonomy**: At minimum: Super Admin, County Chair, Candidate, Campaign Manager, Volunteer Coordinator, Data Entry, Read-Only. Each needs a permission matrix against every ecosystem operation.
- **Multi-Tenant Isolation**: The schema uses `candidate_id` as a soft partition key, but there are no Row-Level Security policies. Must define whether candidates can see each other's donor data, social analytics, etc.
- **Session Management**: JWT vs. session cookies, token expiry, refresh mechanics.
- **OAuth Integration for E19 Social**: The `SocialConfig` class stores `FACEBOOK_APP_ID`, `TWITTER_API_KEY` as env vars, but the OAuth token refresh flow (token exchange, refresh tokens, scope management per platform) is completely absent.[^7_4]

***

## 3. DATABASE SCHEMA CONSOLIDATION (Critical)

The `/database/schemas/` directory contains 60+ SQL files with significant overlap and no migration order:[^7_5]

**Definitions needed:**

- **Canonical Migration Sequence**: Files like `001_broyhillgop_complete.sql`, `COMPLETE_ALL_49_ECOSYSTEMS.sql`, `COMPLETE_DATABASE_SCHEMA 2.sql`, and `1 BroyhillGOP- MASTER_DATABASE_SCHEMA.sql` all appear to be competing "master" schemas. Which one is authoritative?[^7_5]
- **Schema Namespace Resolution**: E20's code references `intelligence_brain.nexus_decisions`, implying a PostgreSQL schema namespace, but most ecosystem inline schemas use the default `public` schema. Need formal namespace allocation per ecosystem.[^7_1]
- **Foreign Key Enforcement**: E19's `social_posts.candidate_id` is declared as plain `UUID` with no FK constraint. E02's `donations.donor_id` is `UUID NOT NULL` but also no FK to a `donors` table. Every cross-table reference needs explicit FK definitions.[^7_2][^7_4]
- **Index Strategy Audit**: Many indexes are created on single columns but complex queries (e.g., donor lookup by name + zip in E51) would benefit from composite indexes that don't exist.[^7_1]

## 4. E20 INTELLIGENCE BRAIN: 7-Model Decision Engine (Stub Logic)

The "7 Mathematical Models" that power GO/NO-GO decisions are the conceptual centerpiece but use hardcoded lookup tables rather than actual mathematical models:[^7_1]

**Definitions needed:**

- **Model 1 (Expected ROI)**: Currently a static dictionary mapping trigger types to benefit values. Needs: historical ROI training data, dynamic benefit calculation based on donor lifetime value from E01, and time-decay weighting.[^7_1]
- **Model 2 (Success Probability)**: Static base probabilities per trigger type. Needs: Bayesian update mechanism using historical conversion data from E02 donations and E30 email engagement.
- **Model 3 (Relevance Score)**: Awards +20 for having a `candidate_id` and +15 for `active_campaign`. Needs: NLP relevance scoring using E07 Issue Tracking vocabulary, news sentiment from E42, and temporal relevance decay.
- **Model 5 (Persona Match)**: Checks training sample count. Needs: actual cosine similarity between E48 Communication DNA voice signatures and draft content vectors.
- **Model 7 (Confidence Score)**: Simple additive scoring. Needs: ensemble confidence using variance across Models 1-6, with a proper confidence interval calculation.
- **Composite Score Weights**: Currently hardcoded as `{relevance: 0.2, persona: 0.3, confidence: 0.3, budget: 0.2}`. These weights need a definition of how they were determined and a mechanism for A/B testing alternative weight configurations.

# BROYHILLGOP BRAIN: Mission-Aligned Platform Architecture

## The Two Primary Missions, Stated Clearly

**MISSION 1 (REVENUE):** Maximize total donor contributions to candidate committees over the full election cycle -- both aggregate dollars and donation frequency -- at the highest achievable cost-to-benefit ratio.

**MISSION 2 (VOLUNTEERS):** Recruit energetic volunteers and enthusiastically motivate them to freely contribute time performing tasks that win votes and raise money.

These missions are not independent -- they form a **flywheel**. Volunteers who door-knock generate new donors. Donors who attend events become volunteers. The BRAIN's job is to accelerate this flywheel at minimum cost.

***

## HOW THE EXISTING CODEBASE MAPS TO EACH MISSION

### Mission 1: Donor Revenue Maximization

The codebase has strong architectural foundations here. E01 Donor Intelligence is the most complete ecosystem, featuring a 3D grading system (Amount x Intensity x Level), RFM analysis with 25+ named segments like "Champions" and "Can't Lose," churn prediction, optimal ask amount prediction, best channel prediction, and lifecycle stage management. E02 Donation Processing handles multi-gateway payment (Stripe with WinRed/ActBlue stubs), FEC compliance checking with contribution limit enforcement, recurring donation management with retry automation, and receipt generation. E30 Email Marketing provides the outreach engine with drip sequences, A/B testing, personalization with 20+ variables, and send-time optimization.[^8_1]

**What's working toward Mission 1:**

- E01's `predict_optimal_ask()` calculates the right dollar amount to request based on donation history -- if average is \$200 and max was \$500, it suggests ~\$350[^8_1]
- E01's `predict_best_channel()` routes high-value donors (\$1,000+) to phone calls and segments others by email open rate vs. SMS response rate[^8_1]
- E01's lifecycle stages (Prospect > New Donor > Active > At Risk > Churned) create a state machine for donor journeys[^8_1]
- E02's FEC compliance prevents donations from being rejected by enforcing limits *before* the charge attempt (\$3,300 per election for federal, \$6,400 for NC state)
- E02's recurring donation manager with `[1, 3, 7, 14]` day retry intervals recovers failed charges
- E30's drip sequences can automatically nurture Prospects through to first donation

**What's MISSING for Mission 1 (Critical Gaps):**

1. **Donor Lifetime Value (LTV) Calculation**: E01's `donor_scores` table has a `predicted_lifetime_value` column, but no code calculates it. This is the single most important metric for the mission. LTV should combine: historical total donated, predicted remaining giving capacity (FEC limit minus contributed), predicted donation frequency over remaining campaign months, and churn probability discount.[^8_1]
2. **Upgrade Path Engine**: No system identifies donors ready to increase their giving level. A C-grade donor who just donated twice in 30 days (intensity spike) should automatically trigger an upgraded ask sequence. E01 tracks `growth_score` but nothing acts on it.[^8_1]
3. **Lapsed Donor Reactivation Pipeline**: E01 identifies "At Risk" and "Churned" donors, and the `next_best_actions` table exists, but no code populates it with reactivation recommendations. The `v_at_risk_donors` view joins to `next_best_actions` but that table is always empty.[^8_1]
4. **Cross-Channel Orchestration**: E30 Email, E31 SMS, E32 Phone, E33 Direct Mail all exist as separate ecosystems but share no coordination layer. A donor who received an email appeal today should not also get an SMS appeal today. No suppression or cadence management exists across channels.
5. **Donation Amount Ladder**: The optimal ask logic rounds to nice numbers but has no concept of an upgrade ladder strategy (e.g., \$50 > \$75 > \$100 > \$150 progression across successive asks). The system should track what a donor was *asked for* vs. what they *gave*, not just final amounts.[^8_1]
6. **Attribution/ROI per Touchpoint**: When a donor gives \$500 after receiving an email, seeing a social post, and attending an event, which touchpoint gets credit? No multi-touch attribution model exists. Without this, cost/benefit optimization is impossible because you can't measure which channel produced the revenue.
7. **Peer Influence Mapping**: E39 P2P Fundraising exists but the social graph (who knows whom, who influences whom) is not connected to E01's donor intelligence. The highest-ROI action may be asking a major donor to personally call three friends, not sending 1,000 emails.

***

### Mission 2: Volunteer Recruitment \& Motivation

E05 Volunteer Management is comprehensive with 56 activity types across 10 categories, a 10-tier grading system (V-A+ to V-D-), shift scheduling, hours tracking, team formation, referral tracking, gamification with 15 badges, and leaderboards.[^8_2]

**What's working toward Mission 2:**

- 56 activity types with value multipliers -- door knocking at 1.5x, fundraising event hosting at 3.0x, data entry at 0.8x -- let the system value volunteer time in proportion to mission impact[^8_2]
- Skill multipliers (legal at 3.0x, technology at 2.5x, general at 0.8x) recognize that specialized skills are worth more[^8_2]
- Show rate tracking with reliability classes (highly_reliable, reliable, moderate, unreliable) enables predictive scheduling[^8_2]
- Referral tracking awards bonus value for recruiting new volunteers[^8_2]
- Gamification badges (First Hour, Century Club, Iron Man, Team Builder) create achievement motivation[^8_2]
- The `build_elite_team()` function auto-identifies top performers (85%+ show rate, 10+ hours)[^8_2]

**What's MISSING for Mission 2 (Critical Gaps):**

1. **Recruitment Funnel/Pipeline**: No system tracks the journey from "heard about volunteering" to "signed up" to "showed up for first shift" to "became regular." The `create_volunteer()` function creates a record, but there's no lead nurturing for prospective volunteers. Where are volunteers sourced from? Social media? Events? Door-to-door? The source isn't tracked and can't be optimized.
2. **Motivation \& Retention Intelligence**: E05 grades volunteers and tracks show rates, but has no proactive system to prevent volunteer dropout. An equivalent of E01's churn prediction should exist for volunteers -- if a V-B volunteer hasn't signed up for a shift in 14 days, that's a warning signal. No `volunteer_churn_risk` field exists.
3. **Task-Volunteer Matching Algorithm**: The system records skills and preferred activities, but no matching engine recommends specific shifts to specific volunteers based on their skills, location, availability windows, and past performance. A graphic designer should be auto-offered content creation shifts, not envelope stuffing.
4. **Impact Visibility for Volunteers**: The gamification system awards badges, but volunteers never see the *downstream impact* of their work. "Your 3 hours of phone banking resulted in 2 new \$100 donations." This connection between E05 (volunteer hours) and E02 (donations generated) doesn't exist. Showing volunteers that their effort directly produced results is the most powerful motivator.
5. **Social/Community Layer**: Volunteering is fundamentally social. No team communication tools, no group chat, no shared photo albums from events, no peer recognition ("Thanks from your team captain"). The system tracks individuals but not the social bonds that keep people showing up.
6. **Onboarding Flow**: No defined first-volunteer experience. What training do they receive? What's their first assignment? How quickly do they hear back after signing up? The critical first 72 hours after signup -- when enthusiasm is highest -- has no automated sequence.
7. **Volunteer-to-Donor Conversion**: E05 has a `donor_id` field on the volunteer record, but no systematic process to convert active volunteers into donors. Volunteers who give 20+ hours/month may be the most likely first-time donors because they're deeply committed to the cause.[^8_2]

## COST/BENEFIT PERFORMANCE FRAMEWORK

The codebase has cost tracking infrastructure in E51 Nexus (5-level cost hierarchy, BVA metrics[ from prior session]) but lacks a unified framework tied to the two missions. Here's what needs to be defined:

**Revenue Side Metrics (Mission 1):**

- Cost Per Dollar Raised (CPDR) = total outreach cost / total donations received
- Donor Acquisition Cost (DAC) = cost to acquire one new donor through each channel
- Donor Retention Rate = percentage of donors who give again within cycle
- Revenue Per Contact = total revenue / total outreach touches
- Recurring Revenue Growth Rate = month-over-month change in active recurring donations

**Volunteer Side Metrics (Mission 2):**

- Cost Per Volunteer Recruited = total recruitment spend / new volunteers signed up
- Volunteer Activation Rate = percentage who complete first shift within 14 days of signup
- Volunteer Retention Rate (30/60/90-day) = percentage still active after N days
- Hours Generated Per Dollar Spent = total volunteer hours / volunteer program cost
- Volunteer-Generated Revenue = donations directly attributed to volunteer activities (door-knock contacts who later donated, phone bank pledges converted)

**Flywheel Metrics (Both Missions):**

- Volunteer-to-Donor Conversion Rate
- Donor-to-Volunteer Conversion Rate
- Network Multiplication Factor = (donors recruited by volunteers + volunteers recruited by donors) / (original contacts)

**The Critical Missing Piece: Attribution Tracking**

E51 Nexus tracks AI API costs per operation[ from prior session], but there's no equivalent tracking of *outcomes per cost*. When the system spends \$0.025 on an AI-generated draft (E51's `COSTS` dictionary), did that draft lead to a published post that drove engagement that converted a donor? The cost side is tracked; the benefit side is not connected.

## 5. MISSION A ENGINE: Donor Revenue Maximization

This is the primary revenue-generating intelligence system, spanning multiple ecosystems working in concert.

### 5.1 The 3D Donor Grading System (E01)

The core scoring engine evaluates every donor on three axes simultaneously:[^9_1][^9_2]

**Axis 1 — Amount (What they give):** Logarithmic grading from F to A++. With your actual donor data, the engine should recalibrate: if your median major donor gives \$15K–\$25K, the A++ threshold at \$10K may need to shift to \$25K+ to properly stratify your high-end segments.

**Axis 2 — Intensity (How engaged they are, 1-10):** Weighted composite of frequency (30%), recency (25%), consistency (25%), and growth trajectory (20%). A donor giving \$500/month with increasing amounts scores higher in intensity than one who gave \$5,000 once two years ago.

**Axis 3 — Level (Where they give: Federal/State/Local):** Critical for multi-candidate operations — routes donors to the candidates where they have demonstrated giving preference.

**Composite Grade Format:** "A+/8/F" means A+ amount, intensity 8/10, Federal-level preference. This three-dimensional notation is what the Brain uses for all decision-routing.

### 5.2 ML-Powered Predictive Analytics (E01 + E21 + E13)

**Churn Prediction:** Identifies donors at risk of lapsing. Uses days-since-last, frequency, amount trend, and engagement metrics. The heuristic model (when ML models haven't been trained on your data) assigns 0.9 risk to 365+ day lapsed donors, adjusting for frequency.[^9_2]

**Optimal Ask Amount:** Calculates the right dollar amount to request on the next solicitation. Uses average donation, maximum historical gift, and growth trajectory. For your high-value segments, this means the model should suggest \$5K–\$25K asks for donors whose history supports it — not the \$50–\$500 range that generic models default to.

**Best Channel Prediction:** Routes each donor to the communication channel where they're most likely to convert: phone for \$1K+ donors, SMS for high-response-rate donors, email as default, in-person for VIPs.[^9_2]

**Lifetime Value Prediction:** Projects total expected revenue from each donor across the remainder of the campaign cycle, enabling prioritization of high-LTV retention actions over low-LTV acquisition.

### 5.3 RFM Analysis (E01)

Classic Recency-Frequency-Monetary scoring with 13 named segments:[^9_2]

- **Champions (555):** Recent, frequent, high-value — protect at all costs
- **Can't Lose (155/154/144):** High monetary but lapsing — urgent rescue protocols
- **At Risk (211/212/221):** Were valuable but disengaging — reactivation campaigns
- **Lost (111/112):** Churned low-value — deprioritize spend


### 5.4 Grade Enforcement System (E01)

Prevents costly mistakes by blocking outreach to donors graded D, F, or U (unknown). This alone saves \$5K–\$50K per campaign by preventing wasted calls, expensive mail pieces, and SMS to disengaged contacts. Restricted grades (C) require manager approval before any outreach.[^9_2]

You're absolutely right, and this is a fundamental design flaw in the current E01 Donor Intelligence System. Let me walk through exactly where and why it breaks.

## The Exact Problem in the Code

The current `AMOUNT_GRADES` configuration in E01 uses hard step-function cutoffs:[^10_1]

```
'A++': 10000,   # $10,000+
'A+':  5000,    # $5,000-9,999
'A':   2500,    # $2,500-4,999
'B':   1000,    # $1,000-2,499
'C':   500,     # $500-999
'D':   100,     # $100-499
'F':   0        # $0-99
```

The `calculate_amount_grade` method iterates through these in order and the first threshold the donor meets becomes their grade. So your \$9,985 donor gets grade **A+** and your \$10,000 donor gets **A++**. These are treated as categorically different human beings by every downstream system.[^10_1]

But it gets worse. The **composite score** formula amplifies this arbitrary cliff:[^10_1]

```
grade_values = {'A++': 60, 'A+': 55, 'A': 50, 'B': 40, 'C': 30, 'D': 20, 'F': 10}
```

That \$15 difference in donations translates to a **5-point gap** in composite score (60 vs 55), which is the same gap as between a \$5,000 donor and a \$2,500 donor. And since amount grade accounts for **60% of the composite**, this cliff echoes through every decision the BRAIN makes — channel routing, event invitations, next-best-action recommendations, everything.

Then the `v_top_donors` view makes it even more damaging:[^10_1]

```sql
WHERE ds.amount_grade IN ('A++', 'A+', 'A')
```

If you used this view to build an event invite list for a "top donor reception" — both the \$9,985 and \$10,000 donor would appear. But if someone coded an "A++ exclusive" event filter? Your \$9,985 donor — who may have given consistently for 8 years, attended every event, recruited other donors — is invisible. Excluded by \$15.

## Why This Is Indefensible at Your Data Scale

With 350,000+ donor histories spanning 10 years across multiple candidates, many at the \$10K–\$50K+ cumulative level, you'll have **hundreds of donors clustered near every threshold boundary**. The \$10,000 line isn't special — it's arbitrary. Consider:

- **Donor A:** \$9,985 total, 47 donations over 8 years, gave last week, increasing trajectory, recruits other donors, attends every event. Grade: A+.
- **Donor B:** \$10,015 total, 1 donation 11 months ago from a cold solicitation, never opened an email since. Grade: A++.

Under the current system, Donor B outranks Donor A on the amount dimension. If anyone builds an event filter, a call list, or a VIP recognition program around the A++ label, Donor A is excluded. You cannot justify that to Donor A. You cannot justify it to your campaign manager. You cannot justify it to anyone who looks at it for five seconds.

And the problem repeats at **every single boundary**: \$4,999 vs \$5,000. \$2,499 vs \$2,500. \$999 vs \$1,000. \$499 vs \$500. Every cliff creates the same absurdity with different dollar amounts.

## The Deeper Structural Problem: Labels Are Not Scores

The real architectural mistake isn't the specific thresholds — it's using **categorical labels as the primary decision object**. The moment you stamp "A+" on someone and route downstream systems based on that string, you've destroyed the continuous information that the actual dollar figure, the donation trajectory, and the 2,200 DataTrust variables contain.

The current system already stores `amount_percentile` (0–1000 scale) and `composite_score` (0–100), which are continuous. But then the code throws those away and gates decisions on the letter grade. The percentile calculation is actually reasonable — it uses a logarithmic scale. A \$9,985 donor gets percentile ~799. A \$10,000 donor gets percentile ~800. One point apart. That's proportional. That's honest. But the system doesn't USE it for decisions — it uses the letter.[^10_1]

## What the Architecture Should Do Instead

**1. Eliminate hard thresholds from all decision-gating.** No event, no call list, no mailing, no VIP program should ever use `amount_grade IN ('A++')` as a filter. Instead, every action should have a **minimum composite score** or **minimum percentile** — continuous values where \$9,985 and \$10,000 are neighbors, not strangers.

**2. Replace the step function with a continuous curve for amount scoring.** The logarithmic percentile already exists in the code. Use it as the primary amount dimension, not the letter. The letter grades can remain as **human-readable labels for display purposes only** — the way ZIP codes describe neighborhoods but don't determine whether you can vote.

**3. Event qualification should use multi-dimensional eligibility, not single-axis cutoffs.** An "elite donor reception" shouldn't be gated by `total_donated >= $10,000`. It should be gated by a composite that weighs:

- Total donated (continuous, not bucketed)
- Donation frequency and recency
- Engagement signals (event attendance history, email opens, call responsiveness)
- Growth trajectory (are they accelerating?)
- Network influence (do they bring other donors?)
- Lifetime value prediction

Under this model, your \$9,985 donor with 8 years of consistent giving, high attendance, and a rising trajectory **scores higher** than the \$10,015 one-time donor — because she IS more valuable to the campaign, and everyone in the room knows it.

**4. Use fuzzy boundaries with overlap zones.** If labels are kept for operational convenience, the boundaries should overlap. A donor at \$9,500–\$10,500 should be flagged as "A++/A+ borderline" and included in BOTH pools for any action. The system should over-include at boundaries, not under-include. The cost of inviting a \$9,985 donor to the A++ reception is zero. The cost of excluding her — insulting a loyal, high-frequency supporter — could be losing \$50,000+ in future donations.

**5. The Grade Enforcement system should NEVER block based on amount alone.** The current code blocks grades D, F, and U from all outreach. But a D-grade donor (\$100–\$499) who gave \$100 last week, opened every email, and just signed up to volunteer is categorically different from a D-grade donor who gave \$100 four years ago and has been unresponsive since. The enforcement should block based on engagement composite — not on what is essentially a wealth proxy.[^10_1]

The bottom line: the BRAIN has 19.8 billion data cells about 9 million people. Using 7 letter grades to make exclusion decisions about those people is like having a telescope that can see galaxies and choosing to sort the sky into "bright" and "dark."

The replacement for E21's 9-feature clustering isn't just "add more features." It's a fundamentally different approach:

**1. **2. What you need is supervised segmentation: train models where the target variable is donation behavior (gave/didn't, amount, frequency, channel, response to specific appeal types) and the features are the 2,200 DataTrust variables. **3. Dynamic microsegment discovery per campaign context.** The microsegments that matter for a US Senate race are different from those that matter for a Yadkin County commissioner race. The engine should re-discover segments for each campaign context — using the contextual scoring framework from the previous discussion — so that the "Rural Gun-Owner Church-Goer" microsegment is activated when it's relevant and dormant when it's not.

**4. Message-microsegment matching.** Every microsegment should have associated message themes, channel preferences, ask amounts, and timing patterns — all learned from historical response data across your 350K+ donor records and 10 years of history. The 46 issue-position scores become the bridge between "who is this person" and "what message moves them."

**5. Capacity vs. behavior separation.** The DataTrust economic variables (income, home value, credit score, charitable giving) measure CAPACITY. The donation history measures BEHAVIOR. These must be modeled separately. A high-capacity, low-behavior donor is a prospect. A low-capacity, high-behavior donor is a loyalist. The microsegment engine needs to understand both dimensions simultaneously and recommend different strategies for each quadrant. The current system conflates them by grading primarily on amount — which is an outcome, not a predictor.

The bottom line: you're sitting on one of the richest political datasets in any state — 9 million voters, 2,200 variables each, 350K donor histories over a decade, 46 issue-position models. **Demographic Match (15%):** Does the voter share the candidate's district demographic profile? A rural, blue-collar, non-college voter in a district that's 68% rural has high demographic affinity. The BRAIN is **not** a single module. It is the **orchestration layer** (E20 Intelligence Brain) that coordinates 43+ ecosystems, making sub-second GO/NO-GO decisions across 905 automated triggers. But the current E20 implementation has fundamental flaws that the redesigned architecture must correct.[^19_1]

### The Core Decision Loop

```
SENSE → SCORE → DECIDE → ROUTE → EXECUTE → LEARN
  │        │        │        │         │        │
  E42      E01      E20      E48       E30/     E21
  News     Donor    Brain    Comm      E31/     ML
  Intel    Intel    Core     DNA       E05      Cluster
```

**SENSE**: News intelligence (E42) monitors 13,000+ sources across all 100 NC counties, detecting crisis events in under 5 minutes. Calendar triggers fire at election milestones (30-day, 7-day, election day). Donation events, engagement signals, and budget thresholds all feed the event queue.[^19_2]

**SCORE**: This is where the existing E01 system needs the deepest correction. The current 3D grading (Amount × Intensity × Level) uses hard thresholds that create cliff effects — a donor at \$999 gets grade C while \$1,000 gets grade B. The redesigned BRAIN replaces these with continuous scoring functions.[^19_3]

**DECIDE**: The E20 engine calculates a decision score (0-100) across five dimensions — urgency, relevance, budget availability, fatigue check, and historical performance — then issues GO, REVIEW, or NO-GO.[^19_1]

**ROUTE**: Communication DNA (E48) applies the candidate's authentic voice profile across every channel, adjusting tone by issue, channel, and audience segment.[^19_4]

**EXECUTE**: Messages deploy through email (E30), SMS (E31), phone/RVM, direct mail, social media, and events (E05 volunteer coordination).

**LEARN**: ML Clustering (E21) runs K-means segmentation using RFM features, tracks segment migration, and feeds performance data back to the scoring engine.[^19_5]

## III. THE SEVEN CRITICAL CORRECTIONS

Based on analysis of the actual code, these are the structural flaws the redesigned BRAIN must fix:

### Correction 1: Continuous Scoring Replaces Hard Thresholds

**Current flaw** in E01: `AMOUNT_GRADES` uses step functions — `'A++': 10000, 'A+': 5000, 'A': 2500`. A donor who gave \$9,999 across 50 transactions gets the same grade as someone who gave \$5,000 once.[^19_3]

**Redesigned approach**: Logarithmic continuous scoring where every dollar shifts the score. The E01 code already hints at this with `amount_percentile` using `math.log10(total_donated + 1) * 200`, but the letter grade still overrides it in enforcement decisions. The BRAIN should use the percentile as the primary signal, with letter grades only for human-readable display.[^19_3]

### Correction 2: Context-Aware Per-Office-Level Scoring

**Current flaw**: E01 treats level preference as a single character — F, S, or L — determined by where the donor has historically given the most money. This collapses a rich giving portfolio into one letter.[^19_3]

**Redesigned approach**: Each donor maintains a **giving portfolio** — their full distribution across all 50+ office types defined in E03. When a candidate for State Senator needs donors, the BRAIN doesn't just look for "S" donors — it scores every donor's affinity for that specific office type, district, and faction alignment. A donor who gave \$20K to federal races but also \$3K to state senate races in the same region is a high-affinity target for a state senate candidate, even though their "level" is "F."[^19_6]

### Correction 3: Office-Level Decision Modes

**Current flaw**: E20's decision engine treats all campaigns identically. The urgency calculation, channel selection, and budget allocation don't differentiate between a US Senate race and a soil \& water conservation district race.[^19_1]

**Redesigned approach**: The BRAIN operates in distinct **decision modes** per office level. E03 defines 50+ office types across federal, state, and local tiers. Each tier has fundamentally different:[^19_6]

- **Dollar scale**: Federal races involve donors giving tens of thousands; local races operate on hundreds
- **Channel mix**: Federal = digital + TV + major donor events; Local = door-to-door + community events + direct mail
- **Urgency curves**: Federal races have 2-year cycles with predictable escalation; local special elections can appear in weeks
- **Volunteer deployment**: Federal = phone banks + digital; Local = yard signs + community canvassing


### Correction 4: Microsegmentation from 2,200 Variables

**Current flaw**: E21 clusters on ~9 features — total_donated, donation_count, avg_donation, days_since_last, frequency_score, monetary_score, recency_score, engagement_score, intensity_score. With 2,200 data variables appended to each of 9 million NC voters, this leaves massive analytical power on the table.[^19_5]

**Redesigned approach**: The BRAIN's segmentation engine should operate in layers:

- **Layer 1 (Identity)**: Demographics, geography, household composition from the voter file
- **Layer 2 (Behavior)**: Full donation history across 10 years to multiple candidates — not just totals but patterns, timing, response to events
- **Layer 3 (Affinity)**: Issue positions, faction alignment (13 factions from E03), candidate archetype match[^19_6]
- **Layer 4 (Propensity)**: ML-predicted scores for donation likelihood, upgrade potential, churn risk, volunteer conversion

The 350,000-donor history contains segments giving tens of thousands, not hundreds. The BRAIN must dynamically calculate dollar-appropriate ask amounts from actual historical records per office and district, not from hardcoded heuristics like `if total >= 1000: return PHONE`.[^19_3]

### Correction 5: Candidate-Voter Affinity Matching

**Current flaw**: E03 stores 273 fields per candidate — faction, issue positions, endorsements, district demographics — but this data is never cross-referenced against donor profiles to compute match scores.[^19_6]

**Redesigned approach**: The BRAIN computes a **Candidate-Voter Affinity Score** by comparing:

- Candidate's faction (13 options: MAGA, Establishment, Liberty, Social Conservative, etc.) against donor's historical support for candidates of each faction type[^19_6]
- Candidate's 60 issue positions against the voter's inferred issue priorities from their giving patterns and appended data[^19_6]
- Candidate's district demographics against the donor's geographic and socioeconomic profile
- Candidate's archetype (incumbent defender, challenger insurgent, open-seat builder) against donor's historical pattern of supporting similar candidate types


### Correction 6: Donor Identity as Giving Portfolio

**Current flaw**: E01 computes a single composite score per donor-candidate pair. This means a donor who supports 8 different candidates gets 8 separate scores with no holistic view.[^19_3]

**Redesigned approach**: The BRAIN maintains a **Donor Identity Profile** that captures:

- **Total giving portfolio**: All candidates supported, amounts, timing, and office levels — spanning the full 10 years of historical data
- **Capacity signal**: Derived from actual maximum gifts, not just appended wealth data. A donor who gave \$25,000 to a congressional race has demonstrated capacity, regardless of what a wealth append says
- **Cadence pattern**: Monthly sustainers vs. deadline-responsive vs. event-triggered vs. crisis-responsive
- **Influence multiplier**: Donors who also volunteer, host events, or recruit other donors amplify their value beyond raw dollars

## CORRECTION 8: FATIGUE MONITORING — THE ANGER PREVENTION SYSTEM

### What the Code Does Now (And Why It Fails)

E20's fatigue system is a blunt instrument. The `contact_fatigue` table tracks three counters — `contacts_today`, `contacts_this_week`, `contacts_this_month` — and the decision engine checks a single rule: if more than 50% of target contacts have hit `MAX_DAILY_CONTACTS = 3`, the campaign proceeds anyway. The `FATIGUE_DAYS = 7` config variable exists but is never actually referenced in the decision logic.[^20_1]

E30 (Email Marketing) has zero fatigue awareness — it checks bounce lists and unsubscribe lists before sending, but has no concept of "this person has received 4 emails this week from 3 different candidates". It will happily send to a donor who just got an RVM, a text, and two other emails the same day.[^20_1]

The fundamental problem: **fatigue isn't a counter — it's a psychological state.** A donor who received 3 contacts this week but opened all 3 and donated from the last one is not fatigued. A donor who received 2 contacts, opened neither, and hasn't given in 6 months is one more touch away from permanent anger.

### The Redesigned Fatigue Intelligence System

The BRAIN must replace the flat counter model with a **multi-dimensional fatigue state machine** that predicts the *emotional trajectory* of each donor:

**Dimension 1: Contact Velocity (What's Happening Right Now)**
Not just "contacts today" but the *acceleration* of outreach. If a donor went from 1 contact/week to 5 contacts/week because election day is approaching, the BRAIN needs to detect the spike relative to that donor's baseline. A monthly sustainer who normally hears from you twice a week can absorb 4 contacts during GOTV week. A once-a-year gala donor cannot.

**Dimension 2: Response Signal Decay (Are They Still Listening?)**
Every contact without a response degrades the relationship. The BRAIN should track a rolling signal-to-noise ratio:

- **Positive signals**: opens, clicks, donations, event RSVPs, volunteer signups
- **Negative signals**: ignored emails, SMS with no click, unanswered calls, unsubscribe-adjacent behavior (opening but never clicking, marking as low priority)
- **Danger signals**: complaint reports from E30's bounce management, email clients flagging as spam, opt-out of one channel (unsubscribed from SMS but still on email — this means "I'm annoyed, not gone")

When the response signal decays below a threshold, the BRAIN should automatically downshift that donor into a **cooling period** — not a hard block, but a reduced cadence. The current binary (contact or don't contact) misses the entire middle ground where most donor anger actually builds.

**Dimension 3: Cross-Candidate Fatigue (The Multi-Campaign Problem)**
With 350,000 donors supporting multiple candidates across 50+ office types, a single donor might be in the outreach queue for a US Senate candidate, a state representative, a county commissioner, AND a school board race — simultaneously. The current E20 fatigue table tracks `contact_id × channel` but not `contact_id × candidate_id`. The BRAIN must maintain a **unified fatigue ledger** across ALL candidates so that the school board race doesn't send the email that pushes a major federal donor over the edge.[^20_1]

**Dimension 4: Emotional Context Sensitivity**
Fatigue tolerance is not constant — it shifts based on external events. During a crisis (opponent attack, breaking scandal), donors EXPECT to hear from the campaign. The BRAIN should adjust fatigue thresholds dynamically:

- **Crisis mode** (E42 alert level = crisis): Fatigue limits temporarily expand by 50% for high-affinity donors, but ONLY for donors whose past behavior shows they respond to crisis appeals
- **Deadline proximity**: FEC filing deadlines and election day create natural urgency that donors understand — slightly higher contact rates are tolerable in the final 72 hours, but ONLY if the donor has been responsive in the past 30 days
- **Post-donation cooldown**: After a donor gives, the BRAIN should enforce a hard 48-72 hour silence (except a thank-you), not immediately ask for more. The current code has no post-donation suppression logic at all

**Dimension 5: Channel-Specific Fatigue Ceilings**
Each channel has a different irritation threshold. The current code applies the same `MAX_DAILY_CONTACTS = 3` regardless of channel. The BRAIN should enforce distinct ceilings:[^20_1]

- **Email**: Highest tolerance — 3-4/week for engaged donors, 1/week for passive
- **SMS**: Most personal, lowest tolerance — 1-2/week max, and ONLY if they've opted in AND responded to at least one in the last 30 days
- **Phone/RVM**: Extremely intrusive — 1 per 2 weeks for mid-tier donors, 1/week for major donors who have a phone relationship
- **Direct mail**: Slow cadence, high tolerance — 2-3/month, but expensive so budget constrains this naturally
- **Social**: Unlimited impressions (they see it if they follow), but direct messages have SMS-level sensitivity

**The Anger Tripwire System**
The BRAIN needs an explicit anger detection layer that triggers BEFORE the donor unsubscribes or stops giving. Warning signals to watch:

- Two consecutive sends with no open → yellow flag
- Three consecutive sends with no open → orange flag, reduce cadence by 50%
- Email opened but immediate unsubscribe click (even if they don't complete it) → red flag, pause for 14 days
- Donor who previously responded within 24 hours now takes 5+ days → trajectory decay, reduce intensity
- Spam complaint from any provider (captured in E30's `email_bounces` table with `bounce_type = 'complaint'`) → IMMEDIATE 30-day hard pause across ALL channels[^20_1]

The goal: **no donor should ever feel harassed.** The cost of one angry donor who badmouths the campaign to their social network — especially a high-capacity donor who gives tens of thousands — far exceeds the revenue from one more ask.

## CORRECTION 9: ISSUE-ADAPTIVE COMMUNICATION — DEFINITIVE WITHOUT ALIENATING

### What the Code Does Now (And Why It Fails)

E48 (Communication DNA) defines `IssueToneProfile` with a hardcoded primary tone and intensity per issue. For taxes, the tone is `WARM_AUTHORITATIVE` at intensity 55 with preferred keywords like "your money" and "family." For crime, it shifts to `PASSIONATE_FIGHTER` at intensity 75.[^20_2]

The problem: **this is candidate-centric, not audience-aware.** The candidate speaks the same way about abortion whether they're talking to a committed pro-life donor or a suburban moderate who leans pro-choice but gives to Republicans on economic issues. E48's `AudiencePreset` only adjusts intensity by a flat offset — seniors get -15, young voters get +10 — but it never changes the *substance* or *framing* of the position based on the audience's own stance.[^20_2]

E03 stores 60 issue positions as simple `support/oppose/mixed` values. The candidate is either for or against something. But communication strategy for hot-button issues requires far more nuance than a database flag.

### The Redesigned Issue-Adaptive Communication System

The BRAIN must create a **three-layer communication strategy** for every hot-button issue that lets the candidate be definitive AND inclusive:

**Layer 1: The Candidate's True Position (Immovable)**
The candidate's actual stance — pulled from E03's `issue_positions` JSONB field — never changes. If the candidate is pro-life, that's the position. If they support constitutional carry, that's the position. The BRAIN never fabricates or softens the candidate's actual beliefs. Authenticity is the E48 Communication DNA's entire purpose.[^20_2]

**Layer 2: The Framing Matrix (Audience-Adaptive)**
The same position can be communicated through different frames depending on who's listening. This is where the BRAIN adds the intelligence that E48 currently lacks:

*Example — Candidate supports school choice:*


| Donor Microsegment | Frame | Language | What to Emphasize | What to Avoid |
| :-- | :-- | :-- | :-- | :-- |
| Social Conservative (faith community) | Religious liberty | "Parents should choose schools aligned with their values" | Faith-based education options | Government funding mechanics |
| Fiscal Conservative | Efficiency | "Competition improves outcomes, lowers costs per student" | Accountability, results data | Cultural arguments |
| Suburban Parent | Pragmatic choice | "Every child deserves access to the best school for them" | Quality, waitlists, practical access | Ideological framing |
| Undecided/Moderate | Common ground | "We all want great schools — let's give families more options" | Bipartisan examples, local success stories | Divisive rhetoric, culture war language |
| Rural donor | Local control | "Rural communities should decide what works for their kids" | One-size-fits-all federal failures | Urban-centric examples |

The candidate's position is the same in every row — they support school choice. But the *entry point*, the *emotional resonance*, and the *vocabulary* adapt to what the recipient actually cares about.

**Layer 3: The Undecided Donor Protocol (The Critical Innovation)**

This is what the current codebase has no concept of. Among the 9 million voter records with 2,200 appended variables, there is a substantial population of donors who:

- Give to Republican candidates on economic or security issues
- Are genuinely conflicted on social hot-buttons (abortion, gender ideology, marijuana)
- Will stop giving entirely if they feel the candidate is *only* about the issue that divides them

The BRAIN must classify each donor not just by their giving history but by their **issue sensitivity profile** — derived from:

- Which candidate archetypes they've supported (E03 faction alignment)
- Which fundraising appeals they've responded to (A/B test data from E30 showing which subject lines and CTAs drove their donations)[^20_1]
- Which issues they've engaged with (email click data on issue-specific links)
- Their appended data signals (organizational memberships, consumer behavior, etc.)

For donors identified as **undecided or cross-pressured** on a hot-button issue, the BRAIN applies the **Undecided Donor Protocol**:

1. **Lead with common ground**: The first contact about a hot-button issue should emphasize shared values, not the divisive conclusion. "We both want to protect families" before "that's why I support X."
2. **Acknowledge complexity**: E48's tone profiles should include a `RESPECTFUL_ACKNOWLEDGING` tone specifically for cross-pressured audiences — distinct from `PASSIONATE_FIGHTER` which works for the base but alienates the middle. Language like "reasonable people can disagree" or "I understand this is personal for many families" doesn't weaken the position — it signals emotional intelligence.
3. **Sequence from safe to sensitive**: When a drip sequence (E30's drip engine) includes a hot-button issue, the BRAIN should ensure the donor has already received and engaged with 2-3 messages on safe-ground issues (economy, infrastructure, veterans) BEFORE the hot-button appeal arrives. Building relationship equity before spending it.[^20_1]
4. **Never lead with the wedge for undecideds**: If the BRAIN's microsegmentation identifies a donor as economically motivated but socially moderate, campaign emails to that donor should lead with economic messaging 80%+ of the time. The candidate can still mention their social positions — but as a secondary frame, not the headline. The existing E48 `get_parameters` method picks a single tone based on the issue being discussed — the redesigned version should pick the tone based on the *intersection* of issue + audience sensitivity.[^20_2]
5. **Monitor for alienation signals**: If an undecided donor receives a hot-button message and subsequently:
    - Opens but doesn't click (read it, didn't engage)
    - Opens the unsubscribe link preview but doesn't complete
    - Goes silent for 30+ days after previously responding within a week

...the BRAIN should immediately pivot that donor back to common-ground messaging and flag them for the anger prevention system described in Correction 8.

**How E48 Communication DNA Must Evolve**

The current `IssueToneProfile` dataclass stores one tone per issue:[^20_2]

```
"taxes": IssueToneProfile("taxes", PrimaryTone.WARM_AUTHORITATIVE, 55, ...)
```

The redesigned version should store a **tone matrix** per issue:

```
"taxes": {
    "base_committed":    ToneProfile(PASSIONATE_FIGHTER, 75, keywords=[...]),
    "fiscal_conservative": ToneProfile(STRONG_CONFIDENT, 60, keywords=[...]),
    "undecided_economic": ToneProfile(CALM_REASONED, 40, keywords=[...]),
    "cross_pressured":   ToneProfile(WARM_AUTHORITATIVE, 45, keywords=[...])
}
```

And for truly divisive issues like abortion or gun policy:

```
"pro_life": {
    "social_conservative": ToneProfile(PASSIONATE_FIGHTER, 80, keywords=["protect life", "stand firm"]),
    "suburban_moderate":   ToneProfile(WARM_AUTHORITATIVE, 40, keywords=["compassion", "support women AND babies"]),
    "undecided":           ToneProfile(CALM_REASONED, 30, keywords=["thoughtful", "deeply personal", "my conviction"]),

That `None` is critical. For cross-pressured donors, some issues should simply not be the subject line, the CTA, or the primary frame of ANY communication. The candidate still holds the position — it's on their website, in their debate answers, in their E03 profile. But the BRAIN's job is to maximize donations, not to broadcast every position to every person. The most effective communication is the one that connects each donor to the candidate through the issues they ALREADY agree on, building enough loyalty and trust that the hot-button differences don't become dealbreakers.

### Where Fatigue and Tone Intersect

These two systems are deeply connected. When the BRAIN detects rising fatigue (Correction 8) AND the recent messages have been on hot-button topics, the risk multiplies. A donor who's annoyed about contact frequency AND conflicted about the issue content is on the edge of permanent loss.

The combined intelligence should work like this:

- **Low fatigue + aligned on issue** → Full intensity, definitive position, passionate tone
- **Low fatigue + undecided on issue** → Moderate intensity, framed position, respectful tone
- **Rising fatigue + aligned on issue** → Reduce frequency but maintain message strength when you do reach out
- **Rising fatigue + undecided on issue** → MAXIMUM CAUTION — drop to minimum cadence, revert to safe-ground issues only, and if the next contact doesn't generate a positive signal, pause outreach for that candidate entirely and let a different candidate in the portfolio re-engage them

The BRAIN's dual mission — maximize total donations AND do so at the highest cost/benefit ratio — requires protecting the long-term relationship. One angry donor lost today costs tens of thousands in lifetime value. The fatigue and tone systems exist to keep the pipeline healthy across the entire election cycle, not just to squeeze out one more ask this week.

## CORRECTION 10: MULTI-CHANNEL ORCHESTRATION — THE BRAIN THINKS IN COMBINATIONS, NOT FUNNELS

You're identifying the deepest structural flaw in the entire codebase. Every channel ecosystem operates as a standalone silo with its own campaign, its own recipient list, its own A/B testing, its own drip sequences, and its own performance tracking — and none of them know the others exist.

**E31 (SMS)** manages `sms_campaigns`, `sms_sends`, `sms_drip_sequences`, `sms_ab_tests` — a completely parallel universe. It checks its own TCPA consent table and has its own shortlink tracking, but doesn't know if the same donor just received an email 10 minutes ago.[^21_1]

But this is a flat list, not a sequenced orchestration. It picks the channels and then hands them off. ### The Multi-Channel Orchestration Engine

**Concept: Campaign Plays, Not Campaign Sends**

Replace the current model — where each ecosystem fires independently — with a **Play** system. A Play is a coordinated sequence of channel touches designed as a unit:

```
PLAY: "Crisis Response - Major Donor"
├── T+0 min:  SMS alert (40 words, urgent tone, donation link)
├── T+15 min: Email (full context, candidate's voice, specific ask amount)
├── T+4 hrs:  IF no email open → second SMS with different angle
├── T+4 hrs:  IF email opened but no click → skip (they're aware, don't push)
├── T+24 hrs: IF donated → thank-you RVM in candidate's actual voice
├── T+24 hrs: IF no action → phone call from finance team (top 200 donors only)
├── T+72 hrs: IF still no action → direct mail with handwritten note feel
└── NEVER:    IF donated at any step → STOP all remaining touches immediately
```

```
PLAY: "Deadline Urgency - Mid-Level Donor"
├── T-7 days:  Email (soft reminder, progress toward goal)
├── T-3 days:  SMS (short, urgent, countdown language)
├── T-1 day:   Email (final push, social proof - "X donors already gave today")
├── T-1 day:   IF email opened + no click → SMS with simplified one-tap link
├── T-0 (deadline): RVM morning-of (candidate voice, personal, grateful)
└── COST CAP:  Total play cost must not exceed projected donation × 0.15
```

```
PLAY: "Lapsed Reactivation - Former A+ Donor"
├── Week 1:    Personalized letter (direct mail, high production value)
├── Week 2:    Email (personal note from candidate, NOT a solicitation)
├── Week 3:    IF email opened → phone call (relationship rebuild, no ask)
├── Week 4:    IF phone answered → email with soft ask, referencing call
├── Week 4:    IF phone not answered → second letter (different angle)
├── Week 6:    IF any positive signal → resume normal cadence
└── Week 6:    IF zero response → park for 6 months, re-evaluate
```

Every step is conditional on the outcomes of prior steps. The BRAIN watches the donor's behavior in real time and branches the play accordingly.

### The Channel Combination Matrix — Cost/Value by Segment

The current E20 code has flat cost-per-contact estimates: `email: $0.01, sms: $0.02, phone: $0.15, direct_mail: $0.75, rvm: $0.03`. This is useless for optimization because it ignores **conversion lift from combinations**.

The BRAIN must learn and maintain a **combination performance matrix**:


| Donor Segment | Email Only | SMS Only | Email→SMS | Email→SMS→Phone | Direct Mail→Email | Optimal Play |
| :-- | :-- | :-- | :-- | :-- | :-- | :-- |
| Elite Federal (A++/10/F) | 2.1% conv | 1.8% conv | 4.7% conv | 8.2% conv | 6.1% conv | Email→SMS→Phone |
| Major State (A+/7/S) | 3.4% conv | 2.9% conv | 5.8% conv | 6.1% conv | 7.2% conv | DM→Email |
| Mid-Level Active | 4.2% conv | 3.1% conv | 6.9% conv | 5.5% conv* | 4.8% conv | Email→SMS |
| Small Dollar Monthly | 5.1% conv | 4.7% conv | 6.2% conv | 3.1% conv** | 2.2% conv | Email→SMS |
| Lapsed 6-12 months | 0.8% conv | 0.6% conv | 1.4% conv | 2.1% conv | 3.8% conv | DM→Email→Phone |
| New First-Time | 6.2% conv | 3.9% conv | 7.1% conv | 4.2% conv** | 5.5% conv | Email→SMS |

*\* Phone calls to mid-level donors have diminishing returns — the cost (\$0.15+ per contact) eats ROI*
*\*\* Phone calls to small-dollar and new donors feel intrusive and generate anger signals*

The BRAIN learns these matrices over time through E20's `brain_learning` table, but the table must be restructured to store **combination sequences** not just individual channels. The current schema records `trigger_category, channel, donor_segment` as independent columns. The redesigned schema needs a `channel_sequence` JSONB column: `["email", "sms", "phone"]` so the learning engine can discover that specific orderings and timings produce specific lift.

### The Cost/Benefit Optimizer

Every play has a cost ceiling derived from the donor's predicted value:

**Rule: The total cost of all touches in a play must not exceed a percentage of the expected return.**

For a donor whose `optimal_ask_amount` (from E01) is \$500 and whose `donation_propensity` (from E21) is 0.35:

- Expected return: \$500 × 0.35 = \$175
- Maximum play cost at 15% efficiency target: \$26.25
- Available budget: Email (\$0.01) + SMS (\$0.02) + RVM (\$0.03) + Phone (\$0.15) = \$0.21 per touch
- That buys up to ~125 touch-equivalents — far more than needed, so this donor can get the full multi-channel play

For a small-dollar donor whose `optimal_ask_amount` is \$25 and `donation_propensity` is 0.20:

- Expected return: \$25 × 0.20 = \$5.00
- Maximum play cost at 15%: \$0.75
- That buys: 1 email (\$0.01) + 1 SMS (\$0.02) = \$0.03 — efficient
- Adding a phone call (\$0.15) pushes to \$0.18 — still within budget but the phone call's conversion lift for small-dollar donors doesn't justify it
- Adding direct mail (\$0.75) EXCEEDS the entire play budget — never do this

This is why one-dimensional thinking destroys ROI. The current E20 code would put a direct mail piece to a \$25 prospect because the event type matched a rule. The BRAIN must dynamically assemble plays where **every channel earns its place through predicted marginal lift relative to marginal cost for that specific donor**.

### Office-Level Play Templates

The channel mix changes drastically by office level, and the codebase currently ignores this entirely:

**Federal Races** (US Senate, Congress) — E03 `office_level = 'federal'`:

- Big money at stake. Major donors expect personal relationships.
- Plays for A++/A+ donors: Direct mail (premium quality) → Personal email from candidate → Phone call from finance director → Event invitation → In-person ask
- Plays for mid-level: Email sequences → SMS urgency → RVM from candidate voice
- Cost tolerance: High. A \$10,000 donor justifies a \$200 multi-channel play over 4 weeks.

**State Races** (Governor, State Senate, State House) — E03 `office_level = 'state'`:

- Mix of major and mid-level donors. Geographic targeting is critical.
- Plays for major donors: Email → Phone → Event invite (county-level)
- Plays for mid-level: Email → SMS → RVM
- Plays for small-dollar: Email only, or Email → SMS for deadline pushes
- Cost tolerance: Moderate. A \$2,500 donor justifies a \$50 play.

**Local Races** (County Commissioner, Sheriff, School Board) — E03 `office_level = 'local'`:

- Small budgets. The \$0.75 direct mail piece might BE the entire campaign budget per donor.
- Plays for everyone: Email + volunteer door-knock (E05 coordination)
- Phone/SMS only for identified supporters within the district
- The volunteer channel REPLACES paid channels — a neighbor knocking on a door converts at 10x the rate of any email, at zero media cost
- Cost tolerance: Very low. A \$100 donor justifies a \$3 play maximum.


### Real-Time Adaptation Within a Play

The BRAIN doesn't just launch a play and let it run. It watches every signal and adapts:

**Signal: Email opened within 2 hours**
→ This donor is attentive right now. If the play called for an SMS follow-up at T+4 hours, accelerate it to T+1 hour while they're still engaged.

**Signal: SMS link clicked but donation page abandoned**
→ Don't send the same ask again. The next touch should acknowledge the interest ("We noticed you started — here's what your \$250 would fund specifically...") through a different channel (email with more detail, or an RVM that adds emotional weight).

**Signal: Phone call answered, donor said "I'll think about it"**
→ Log the verbal signal. Schedule a single follow-up email in 48 hours with a gentle reminder. Do NOT call again unless the donor explicitly asked for a callback. "I'll think about it" is one step from "stop calling me."

**Signal: Direct mail piece returned (bad address)**
→ Update the voter file record. Shift ALL remaining play touches to digital channels. Flag for address verification before next cycle.

**Signal: Donor gives at any point during the play**
→ IMMEDIATELY cancel all remaining solicitation touches. Replace with a thank-you sequence: instant email receipt → thank-you SMS from candidate → handwritten-style thank-you card (for major gifts). The worst possible donor experience is giving \$5,000 and then getting a phone call the next day asking for more.

### The Cross-Candidate Coordination Layer

Here's where it gets complex with 350,000 donors potentially supporting multiple candidates across 50+ office types. The BRAIN must prevent:

**Channel collision**: Donor gets an email from the Senate candidate at 10 AM, an SMS from the Governor candidate at 10:30 AM, an RVM from the State House candidate at 11 AM, and a direct mail piece from the County Commissioner arriving the same day. Each individual play was "correct" — but the combined effect is a donor who feels besieged by Republicans.

**Solution: The Unified Outreach Calendar**

Before any ecosystem fires a touch, it must reserve a slot on the donor's personal outreach calendar:

```
DONOR: John Smith (A+/8/F, federal preference, suburban Forsyth County)

Monday:     [RESERVED] Senate candidate - Email (crisis response play)
Tuesday:    [OPEN]
Wednesday:  [RESERVED] Senate candidate - SMS follow-up (same play, step 2)  
Thursday:   [OPEN]
Friday:     [BLOCKED] Governor candidate wanted to send - DENIED, 2 touches already this week
Saturday:   [OPEN - but reduced priority, weekend send-time data shows lower opens]
Sunday:     [QUIET] No outreach
```

The Governor candidate's play for John Smith gets **deferred, not cancelled** — it moves to the following Tuesday. And the BRAIN logs the deferral reason so the learning engine can factor cross-candidate interference into future play construction.

**Priority arbitration**: When two candidates compete for the same donor's attention in the same week, the BRAIN resolves by:

1. **Election proximity** — the candidate whose election is sooner gets priority
2. **Donor affinity** — the candidate the donor has historically given more to gets priority
3. **Ask urgency** — a crisis response outranks a routine solicitation
4. **Budget efficiency** — the play with the better predicted ROI for this donor gets priority

### How the Learning Engine Evolves Plays

The current `brain_learning` table stores flat metrics: `sends, opens, clicks, conversions, revenue` per category/channel/segment. The redesigned learning engine must store:

- **Play-level performance**: Which complete multi-step sequences produce which conversion rates for which segments
- **Timing discoveries**: The BRAIN might learn that for rural donors over 65, an RVM at 7 AM followed by an email at noon produces 2.3x the conversion of the reverse order — because they listen to voicemail over morning coffee and then check email after lunch
- **Channel synergy multipliers**: Email alone = 3% conversion. SMS alone = 2.5%. Email followed by SMS 4 hours later = 6.8% — the combination isn't additive, it's multiplicative. The BRAIN must detect these synergies
- **Diminishing returns curves**: At what point does adding a 4th touch to a play actually DECREASE conversion because it tips into fatigue? This connects directly to Correction 8's anger prevention system — the multi-channel orchestrator and the fatigue monitor must share state in real time


### The Print Channel — Not an Afterthought

Direct mail is the most expensive per-unit channel but for certain segments it remains the highest-converting — especially for older donors, lapsed major donors, and local races where a physical piece carries community credibility. The BRAIN must treat print not as a standalone campaign but as a component in multi-channel plays:

- **Print as anchor**: For lapsed A++ donors, the premium direct mail piece is the opening touch that reestablishes legitimacy. The email that follows 5 days later references the letter they received. The phone call a week after that references both.
- **Print as closer**: For donors who've engaged digitally but haven't converted, a well-timed direct mail piece with a reply envelope and a handwritten-style note can be the physical commitment device that tips the decision.
- **Print for local races**: When the budget can't afford phone banking and the digital list is thin, a targeted direct mail piece to the 2,000 most likely donors in the district — coordinated with a volunteer door-knock the following week that references the mailer — is the highest-RO

## CORRECTION 16 (REVISED): CANDIDATE MEDIA LIBRARY — A Unification Layer Over Existing Ecosystem Storage, Not a Replacement

**What was wrong with my first draft:** I designed the Vault as if it were a brand-new centralized filing system. Nothing gets moved. Nothing gets duplicated. Every ecosystem keeps writing where it already writes. The Library just knows where everything is.

### 16.2 What's Missing: The Unified Index

Every ecosystem stores its own stuff perfectly. What's missing is a **single window** where the candidate sees ALL of it organized by THEIR mental model (ads, photos, videos, invoices, logs) rather than by ecosystem number. The unification layer is a **read-through virtual catalog** — not a new storage engine.

### 16.3 How the Index Stays Current (Sync, Not Copy)

Every ecosystem already writes to its own tables. The unification layer needs to **know** when new content appears. Two mechanisms:

```

SYNC MECHANISM 1: EVENT BUS LISTENERS
│
│  The platform already has an event bus (multiple ecosystems
│  reference it). When any ecosystem creates, updates, or
│  deletes content, it emits an event. The Library Index
│  subscribes to ALL content-producing events:
│
├── E08 → "content.created", "content.updated", "content.published"

## CORRECTION 17: ADMINISTRATIVE OPERATIONS LAYER — Staff Infrastructure, Campaign Manager Integration \& Candidate Silo Bridge

**What was missing:** The entire architecture up to now has been designed from the **candidate's perspective** — their dashboard, their silo, their creative wizard, their donor data. But behind every candidate silo is a **live administrative staff** that runs the platform itself: accounting, tech support, customer support, a call center with campaign managers assigned to individual candidates. This staff needs its own integrated infrastructure that connects INTO each candidate silo without violating the silo's isolation.

**What already exists that we strengthen (not replace):**

- **E12 Campaign Operations** already has `campaign_team` table (roles, departments, permissions, hourly rates), `campaign_tasks` (assigned, tracked, commented, audited), `campaign_projects` (milestones, budgets), `campaign_calendar` (events, attendees, reminders), `campaign_messages` (internal team messaging), `time_entries` (time tracking), and views for task summaries, team workload, and project progress.[^25_1]
- **E24 Candidate Portal** already has `portal_users` (authentication, roles, permissions per candidate), `portal_alerts` (prioritized, typed, ecosystem-sourced), `portal_approvals` (review queue), `portal_actions` (action log), `portal_saved_reports`.
- **E45 Video Studio** already has Zoom API integration with `schedule_town_hall` method.[^25_2]
- **E54 Calendar Scheduling** already has FullCalendar with drag-drop, color-coded categories, RSVP management, automated reminders (Email/SMS/RVM), conflict detection.[^25_3]
- **E11 Training LMS** already exists for onboarding/training workflows.

**What's missing:** These ecosystems were all built as if the "team" is the candidate's own campaign team. They don't distinguish between **platform administrative staff** (your employees who run the operation) and **candidate-side users**. There's no concept of an admin staff member who can see across multiple candidate silos, no campaign manager assignment model, no admin-side accounting/support/call center infrastructure, and no "Campaign Manager" button in the candidate's nav that bridges the two worlds.

## IV. HOW THE BRAIN EXECUTES MISSION 1 (MAXIMIZE DONATIONS)

The donation maximization loop operates continuously:

### A. The 3D Grading Pipeline (E01)
Every donor in the 350K+ history gets a three-dimensional grade recalculated in real-time:

1. **Amount Dimension** (A++ through F): Where $10K+ = A++, $5K+ = A+, down to F for <$100. Given the actual donor base, many segments cluster at A+/A++ levels with tens of thousands in cumulative giving.

2. **Intensity Dimension** (1-10): Weighted composite of frequency (30%), recency (25%), consistency (25%), and growth trajectory (20%). A donor giving $2,000 quarterly with increasing amounts scores 9-10; a $50K one-time donor who hasn't given in 18 months might score 3-4.

3. **Level Dimension** (F/S/L): Federal, State, or Local preference based on where their donation dollars have historically gone across 10 years of multi-candidate history.

### B. The Decision Loop
When any trigger fires (905 types across news, donations, engagement, calendar, compliance, budget, crisis, GOTV categories):[^27_2]

1. **E20 Brain** scores the event (urgency × relevance × budget availability × fatigue check × historical performance) in under 1 second
2. Score ≥70 = **GO** → channels selected, targets selected, budget allocated
3. Score 50-69 = **REVIEW** → human approval required
4. Score <50 = **NO-GO** → suppressed
5. **Grade Enforcement** blocks any outreach to D/F grades (saves $5K-$50K per campaign in wasted spend)
6. **Fatigue Management** prevents over-contacting (max 3 contacts/day per donor, 7-day minimum between same-channel contacts)

### C. Optimal Ask Calculation
For each donor, E01's ML pipeline calculates:
- **Predicted Next Amount** (based on giving trajectory)
- **Optimal Ask Amount** (calibrated between average and maximum historical gift)
- **Best Channel** (phone for $1K+ donors, SMS for high-response-rate donors, email for steady engagers)
- **Churn Risk** (heuristic + ML ensemble: days since last × frequency × trend)
- **Lifetime Value** projection

### D. Revenue Feedback Loop
Every campaign outcome (sends, opens, clicks, conversions, revenue) is recorded in E20's `brain_learning` table, continuously updating ROI calculations per trigger category, channel, and donor segment. The Brain gets smarter with every cycle.
